Sanjeevini · agent: Jeeva

Bring dead bioinformatics tools back to life.

Jeeva reads a dead research repo, writes its own falsifiable plan, and repairs it inside a Docker sandbox until it provably runs — emitting a pinned, verified, composable brick.

Python ≥ 3.10 Long-read first-class (ONT · PacBio) Nextflow · Snakemake · WDL Typed genomic I/O MCP server 319 tests · 86% coverage

Install

# From source (this repo)
git clone https://github.com/musicofthings/sanjeevini
cd sanjeevini
python -m venv .venv && source .venv/bin/activate
pip install -e .

# Optional extras, opt in to what you need:
pip install -e '.[agent]'     # autonomous resurrect (anthropic; needs Docker + ANTHROPIC_API_KEY)
pip install -e '.[search]'    # semantic registry search (sentence-transformers + faiss)
pip install -e '.[longread]'  # ONT/PacBio raw-signal I/O (pysam, pod5)
pip install -e '.[mcp]'       # MCP server deps

The command is jeeva. Run jeeva --help to list every subcommand.

What each verb does

jeeva pin

Resolve packages to the versions that were live on a date — PyPI, conda/Bioconda, or Bioconductor.

jeeva decay-check

Agent-free: does a repo still install and run today? Emits naive_runs / install_fails / run_fails.

jeeva resurrect

Autonomous repair loop: scout → plan → build → repair → prove → emit a verified contract bundle.

jeeva registry

Browse, search, and pull the catalog of revived tools. Semantic search with a lexical fallback.

jeeva run

Compose revived bricks into a pipeline. Port types are validated before any container starts.

jeeva mcp

Expose every verb as MCP tools so any agent (Claude Code, etc.) can call them directly.

Quickstart

1 · Pin to a commit era

jeeva pin --date 2021-01-01 pysam numpy
# → pip install pysam==0.16.0.1 …
jeeva pin --conda --date 2021-06-01 bwa samtools   # Bioconda/conda-forge
jeeva pin --bioc  --date 2021-06-01 DESeq2          # Bioconductor

2 · Check whether a repo has decayed

jeeva decay-check https://github.com/some/dead-tool --json
# → {"verdict": "install_fails", "stage": "install", "reason": "pip_requirements_failed", "url": "..."}

3 · Resurrect a dead tool

# Needs the [agent] extra, a running Docker daemon, and ANTHROPIC_API_KEY.
export ANTHROPIC_API_KEY=sk-...
jeeva resurrect https://github.com/nanoporetech/medaka \
    --turns 40 --checkpoint-dir ./.jeeva/medaka   # resumable
# On PASS, a verified bundle lands in ./contracts/<tool>/:
#   contract.yaml · predict.py · Dockerfile · smoke_test.sh · REPRODUCE.md · PROVENANCE.json
Jeeva never declares success on its own word: a resurrection only PASSes when a command it marks as the sanity check actually exits 0 inside the sandbox — the measurable, falsifiable criterion from the plan.

4 · Browse & search the registry

jeeva registry list --platform ont
jeeva registry search "SV caller for ONT"
jeeva registry pull sniffles2

5 · Compose a typed pipeline

# Validate port types without running anything:
jeeva run examples/pipelines/ont_sv_calling.yaml --dry-run
# → PASS — 3 steps, all port mappings type-check.

What watching it work taught us

Every mechanism below exists because a real run failed without it.

Working memory

The first PyPore attempt burned 38 turns re-reading the same two files, probing past end-of-file and grepping for a class that does not exist. The agent saw only the last 800 chars of output and nothing from earlier turns. It now writes durable notes replayed into every later prompt, and the loop refuses to re-run an inspection that cannot reveal anything new. The same resurrection then finished in 22 turns.

Repairs ≠ diffs

A successful run once recorded zero bugs fixed, because fixes were only counted when the agent emitted a source patch. Repointing a dead apt mirror or installing a missing compiler produces no diff — yet those are the most reusable findings there are. They now count, and they seed the cross-run lesson store.

Honest recipes

The emitted smoke_test.sh replayed every command including orientation greps, and claimed a git clone made it runnable anywhere while its commands referenced container-absolute paths. It now replays only state-changing steps and guards on the path it genuinely needs.

It gets smarter every run

Each resurrection that repairs something teaches the next one. The loop records every applied patch as a lesson — the traceback it responded to (the symptom), the fix, and the framework — into a store under the cache root. On the next run the agent retrieves the relevant lessons and injects them into its prompt.

Relevance is deliberately dependency-free: a framework match plus keyword overlap between the current traceback and a stored symptom. Enough to surface “last time a TensorFlow 1.x tool hit a missing-AVX abort, a non-AVX build fixed it” exactly when the agent is staring at that traceback — without pulling in an embedding model.

A repair is not only a source edit. Repointing a dead apt mirror, installing a missing compiler, or pinning a dependency to its commit era all fix real decay, and they are the most reusable things a run discovers. The PyPore resurrection produced four such lessons — the first real entries in the store.

Lessons are injected as hints, not instructions. The prompt tells the agent to verify before trusting one, and PASS is still decided only by a real exit code — so a wrong lesson cannot poison a later run.

How Jeeva works

  1. Sandbox — a disposable Docker container with long-read hardware profiles, turn-level checkpoints, and read-only raw-signal mounts.
  2. Pinners — resolve every dependency to its commit era across PyPI, conda/Bioconda, and Bioconductor.
  3. Scouts — read the repo (Python, R/Bioconductor, or a Nextflow/Snakemake/WDL workflow) and write a plan with a falsifiable sanity check.
  4. Repair loop — build → run → read the real traceback → patch → retry, checkpointing each turn; emit a contract on PASS.
  5. Registry — a searchable catalog of every revived tool, with typed contracts and provenance.
  6. Compose — wire bricks into pipelines; genomic I/O types are checked before any container starts.
  7. Learn — every fix is banked as a symptom → fix lesson and retrieved on later runs, so each resurrection starts smarter than the last.
